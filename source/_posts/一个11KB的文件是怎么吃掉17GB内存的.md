晚上九点多，一台 94GB 内存的共享开发机卡成了幻灯片。管理员用 root 登上来，手起刀落杀掉一个进程，机器瞬间复活。群里只留下一个问题：**是谁的进程，把内存打爆了？它在干什么？**

我接下了这个案子。这是一次从"完全没有 root 权限"开始的取证，一路上有四次反转：凶手进程读的文件只有 11KB；用户敲的命令和实际执行的程序不是同一个；本该两分钟就杀掉它的看门人先死了；而理论上的最后防线——内核 OOM killer——从头到尾都没打算出手。

## 一、没有 root 的取证：atop 是行车记录仪

排查 OOM 的标准姿势是看内核日志：

```bash
$ dmesg -T | grep -i "out of memory"
dmesg: read kernel buffer failed: Operation not permitted

$ ls -l /var/log/kern.log
-rw-r----- 1 syslog adm ... /var/log/kern.log   # 只有 adm 组能读
```

普通用户，两条路都堵死。但 `/var/log` 里躺着一个宝藏目录：

```bash
$ ls -la /var/log/atop/
-rw-r--r-- 1 root root 988338718 Jul 21 21:21 atop_20260721
```

**atop 的历史文件是 world-readable 的**，每 10 秒一个采样，记录了每个进程的 CPU、内存、状态、完整命令行——甚至包括已经退出的进程。它就是这台机器的行车记录仪。

第一步，用 `atopsar -m` 拉内存曲线，找"尸体消失的时刻"：

```text
21:18:42    memfree  1636M
21:18:52    memfree  2126M
21:19:02    memfree 20114M    ← 空闲内存 +18GB 跳变
```

21:18:52 到 21:19:02 之间，有个占着约 18GB 的进程死了。死亡时刻锁定。

第二步，回放这个时间窗，找 RSS 匹配的进程：

```bash
$ atop -r /var/log/atop/atop_20260721 -P PRM -b 21:18:40 -e 21:19:10 | ...
21:18:52  PID=1626391  RSS_KB=17729280    # 17.7GB，与跳变吻合
```

第三步，用 PRG 记录拿它的完整命令行和退出状态：

```text
21:18:52  PID 1626391 (claude) 状态 D
  cmdline: ugrep -G --ignore-files --hidden -I ... -o -E
           '.{0,150}(关键词A|关键词B)[^\\]{0,200}' /tmp/xxx/ctx_123.txt

21:19:02  PID 1626391 (claude) 状态 E  exit=265   # 265 & 0x7f = 9，SIGKILL
  父进程 bash 退出码 137                            # 128 + 9，实锤被 kill -9
```

凶手找到了：一条 `ugrep` 正则搜索，被 SIGKILL 处决。它来自一个 AI 编码代理（Claude Code）的批量日志审计任务——在导出的日志文本里检索两个关键词，顺带抓取前后各一两百个字符的上下文。

一个 grep 而已，怎么会吃 17.7GB？

## 二、反转一：文件只有 11KB

我的第一反应是：目标文件一定巨大，或者是个没有换行的超长单行文件，把行缓冲撑爆了。

```bash
$ ls -la /tmp/xxx/ctx_123.txt
-rw-r--r-- 1 ... 11464 ... ctx_123.txt        # 11KB
```

**11KB。** 一个正常人类眼睛一秒扫完的文件。

再看它的内存增长曲线（还是 atop 回放）：

```text
20:01:12  RSS = 1.5 GB     # 启动后 10 秒
20:10:02  RSS = 15.2 GB    # 9 分钟冲到 15GB
20:50:02  RSS = 16.7 GB
21:18:02  RSS = 17.3 GB    # 78 分钟，一个匹配结果都没输出
```

前 9 分钟狂吃 15GB，之后匀速阴燃。这个形状说明它根本没在"搜索"——**它卡在正则编译阶段就没出来过**。问题不在文件，在这条正则本身：

```regex
.{0,150}(关键词A|关键词B)[^\\]{0,200}
```

## 三、反转二：你敲的是 grep，跑的是 ugrep

atop 里有个诡异细节：进程的 comm 是 `claude`，命令行却是 `ugrep`。而审计任务的原始命令，写的明明是 `grep`。三个名字，一个进程。

翻开 Claude Code 会话的 shell 快照，真相是一个影子函数：

```bash
grep () {
    ...
    # 用 claude 二进制内嵌的 ugrep 替身执行
    ( exec -a ugrep "$_cc_bin" -G --ignore-files --hidden -I \
        --exclude-dir=.git ... ${1+"$@"} )
}
```

Claude Code 会在它的 Bash 会话里把 `grep`、`find` 这类命令**影子（shadow）成自带的 ugrep/bfs**——好处是行为跨平台一致、默认忽略 `.git` 等目录；代价是：你以为你在用 GNU grep，实际在用一个**引擎完全不同**的实现。

- **GNU grep**：惰性 DFA + 回溯混合，按需构造状态，有界量词几乎不额外花钱；
- **ugrep**：基于 RE/flex 的 **DFA 全展开**编译。在 Unicode 模式下，`.` 是"任意 UTF-8 字符"这个庞大字符类，`.{0,150}` 意味着把这个字符类**拷贝展开 150 层**再做子集构造，配合后面的 `[^\\]{0,200}` 再来 200 层——状态数组合爆炸。

于是同一条命令，两个世界：

```bash
# GNU grep：0.088 秒，找到 2 处匹配
$ time /usr/bin/grep -c -o -E '.{0,150}(关键词A|关键词B)[^\\]{0,200}' ctx_123.txt
2
real    0m0.088s

# 内嵌 ugrep：限 2GB 虚拟内存复现，20 秒内直接 core dump
$ timeout 20 bash -c 'ulimit -v 2097152; exec -a ugrep <claude二进制> \
    -G -o -E ".{0,150}(关键词A|关键词B)[^\\\\]{0,200}" ctx_123.txt'
timeout: the monitored command dumped core
```

**爆炸只跟正则有关，跟文件大小无关。** 哪怕目标是个空文件，它照样把内存吃穿。

## 四、反转三：看门人先死了

新的疑问：Claude Code 的 Bash 工具是有超时的（默认 2 分钟），到点会杀掉子进程。这条 grep 凭什么跑了 78 分钟？

时间线给出了答案：

```text
20:01:03  grep（ugrep）启动，开始吃内存
20:12     用户的 SSH 会话断开 → claude 进程随之死亡
          → 负责执行超时的"看门人"没了
          → bash + ugrep 变成孤儿（PPID=1）
20:12-21:18  孤儿 ugrep 处于 D 状态（深陷内核内存回收），
             免疫 SIGHUP，无人看管，继续阴燃 66 分钟
```

超时机制是 claude 进程**在用户态自己执行**的。宿主一死，机制归零。而 D 状态的进程连 SIGKILL 都要等它从内核态醒来才处决得掉，SIGHUP 更是挠痒痒。

教训很朴素：**无人值守的批量任务，把宿主进程放进 tmux。** SSH 断开 tmux 里的会话不死，超时、Ctrl-C、事后干预都还在。成本为零。

## 五、反转四：内核 OOM killer 根本没打算出手

最后一个问题：都这样了，内核 OOM killer 为什么不管？

```bash
$ grep oom_kill /proc/vmstat
oom_kill 0        # 开机 25 天，一次都没触发过
```

因为在内核看来，这台机器**从来没有真正 OOM**：空闲内存最低掉到 747MB 时，还有 20 多 GB 的 page cache 可以回收。内核的算盘是"先回收缓存救急"，于是全机进入疯狂直接回收（direct reclaim），15 分钟负载冲到 35，人人卡顿——但 OOM killer 的触发条件始终不满足。

这是共享大内存机器的经典困境：**单个失控进程吃不满整机内存，却能靠制造回收压力把所有人拖下水**，而系统级的最后防线（OOM killer）恰好对这种"半死不活"最无感。最终出手的只能是人肉 root：`last` 记录显示管理员 21:07 登录，21:18:56 手刃了它。

## 六、复盘：三层防线

这次事故是三层漏洞的叠加，每层对应一条防线：

**第 1 层：正则纪律（直接原因）**
- 有界量词 `{m,n}` 上限超过 20 的正则，强制 `command grep` 绕过影子函数走 GNU grep；
- "关键词 ± 上下文"这种需求，两步走：先 `grep -n` 拿行号，再用脚本按偏移切片。别用 `.{0,N}关键词.{0,M}` 一发入魂；
- 预期超过 1 分钟的重型命令，主动包一层 `timeout` + `ulimit -v`。

**第 2 层：孤儿治理（放大器）**
- 长时间跑批的代理会话进 tmux，别让 SSH 断开带走看门人。

**第 3 层：系统兜底（最后防线）**
- 用户级：一个 5 分钟一跑的 cron 巡检——可用内存低于阈值、或单进程 RSS 超阈值就发 IM 告警。这次要是有它，20:10 就该有人收到消息，而不是 78 分钟后靠 root 巡逻；
- root 级：给共享机的 systemd user slice 配 `MemoryHigh`，或者装 earlyoom。让"半死不活"也有人管。

## 尾声

写完巡检脚本上线那天，我给它的告警文案里加了一行处置提示：

> atop -r /var/log/atop/atop_$(date +%Y%m%d) 回放定位

毕竟这次能破案，全靠 atop 这台默默转了 25 天的行车记录仪。没有 root 的时候，它就是你的 root。
