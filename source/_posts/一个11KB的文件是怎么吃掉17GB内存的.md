晚上八点多，我发现自己 SSH 登不上开发机了。

不是拒绝连接，不是密码错误，而是那种最折磨人的状态：TCP 能通，认证能过，然后光标一闪一闪，shell 永远出不来。挂了快十个小时的老会话也断了，重连上去撑了两分钟又被踢下来。

这台开发机是团队共用的，几十个人在上面跑各自的任务，每个用户有内存限额。我第一反应是机器出问题了，但我连上去看一眼的能力都没有——于是我让笔记本上的 Claude 替我 debug："为什么 SSH 登录不上？"

这是一次完整的排查实录，由笔记本上的 Claude（本地视角）和事后开发机上的 Claude（现场取证）接力完成。一路上有五次反转：先查 CPU 却查错了方向；凶手进程读的文件只有 11KB；用户敲的命令和实际执行的程序不是同一个；超时机制明明开了枪却没打中要害；而每用户内存限额这道防线，不但没能阻止灾难，反而正是"机器卡成幻灯片但谁也不死"的根源。

## 一、第一个误导：CPU 看起来的确很忙

断断续续登上去的那几分钟里，最先看到的是这个：

```bash
$ uptime
21:25:38 up 25 days, 19 users, load average: 4.23, 15.18, 34.96
```

15 分钟负载 35，事后回放峰值到过 **56**。再瞟一眼 CPU，内核态占用高得吓人。我的第一判断是：谁在跑重计算任务，CPU 打满了。

这个判断错了，而且错得很有代表性。事后从 atop 历史数据里回放当时的 CPU 分布（这台机器 24 核）：

```text
20:02:42  usr=183%  sys=39%   idle=2174%   ← 正常：用户态干活
20:03:02  usr=95%   sys=809%  idle=1060%   ← 突变开始
20:03:12  usr=56%   sys=1502% idle=830%    ← 15 个核陷在内核态
```

用户态几乎没人干活，**24 个核里有 15 个核在内核态空转**。这不是"计算任务吃 CPU"，这是内核在替某个进程疯狂做内存回收（direct reclaim）。CPU 忙是症状，不是病因。

负载 56 同样是个陷阱：Linux 的 load average 统计的是"可运行 + 不可中断睡眠（D 状态）"的任务数。一台内存告急的机器上，大量进程卡在 D 状态等内存，负载会飙得很高——**而它们一个 CPU 周期都没用**。load 只告诉你"有很多任务在等"，不告诉你在等什么。

还有一个细节值得先说：这次**严格来说不是 OOM**。整机 94GB，事发全程还有约 29GB 的 MemAvailable，没有用 swap，内核 OOM killer 一次都没触发。真正的杀伤形态是"资源风暴"：高内核态 CPU + 大面积 D 状态 + 一个内存异常进程，三者互相喂养。这也是为什么它格外难缠——每一项单看都够不上系统自动处置的门槛。

## 二、PSI：一眼分清到底是哪种资源在卡

资源瓶颈无非几种：CPU、内存、IO。传统工具各管一摊（top 看 CPU、free 看内存、iostat 看 IO），你得挨个看完再拼图。而内核 4.20 之后自带的 **PSI（Pressure Stall Information）** 直接把答案写在三个文件里：

```bash
$ head /proc/pressure/cpu /proc/pressure/memory /proc/pressure/io
```

PSI 统计的不是"资源用了多少"，而是"**任务因为等这种资源，停顿了多长时间**"——直接量化痛苦本身。两行指标：

- `some`：至少有一个任务在等这种资源（部分人难受）；
- `full`：**所有**非 idle 任务都在等（整机瘫痪，谁也干不了活）。

事发时段的 PSI（atop 的历史文件里存了每 10 秒的采样，`atopsar -B` 可以回放）：

```text
时刻       cpusome   memsome  memfull   iosome  iofull
20:02:52       1%        3%       3%      12%     11%
20:03:02       9%       48%      46%     26%     23%
20:03:12      51%       97%      88%      1%      0%
20:04:52      73%       99%      90%      2%      0%
   ……（此后 70 多分钟一直维持在这个水平）……
21:16:42       8%       83%      69%      0%      0%
```

答案一目了然：**memsome 99%、memfull 90%**——几乎全部任务在等内存，机器 90% 的时间处于"所有人都动不了"的状态。cpusome 虽然也有 70%，但 `cpufull` 恒为 0%（CPU 永远有活干，只是干的是回收内存的脏活）；IO 只在最初 10 秒抖了一下（那是内存页被挤出去的余波）。

**内存，是内存。** 如果 20:03 那一刻有人看一眼这三个文件，10 秒就能定位方向，不用在 CPU 上绕弯路。

顺带解释了 SSH 为什么登不上：sshd 本身是 root 的进程活得好好的，但登录成功后，pam_systemd 会把你的会话塞进你自己的 user slice（cgroup）——而那个 cgroup 正处于深度内存节流中（下文会讲）。你的 shell 一出生就要分配内存，一分配就被卡进回收队列。于是"认证能过、shell 出不来"。

## 三、没有 root 的取证：atop 是行车记录仪

方向定了，接下来要找到吃内存的人。排查内存事件的标准姿势是看内核日志，但普通用户两条路都堵死：

```bash
$ dmesg -T | grep -i "out of memory"
dmesg: read kernel buffer failed: Operation not permitted

$ ls -l /var/log/kern.log
-rw-r----- 1 syslog adm ...     # 只有 adm 组能读
```

好在 `/var/log/atop/` 里躺着宝藏：

```bash
$ ls -la /var/log/atop/
-rw-r--r-- 1 root root 988338718 Jul 21 21:21 atop_20260721
```

**atop 的历史文件是 world-readable 的**，每 10 秒一个采样，记录每个进程的 CPU、内存、状态、完整命令行——连已经退出的进程都有。它就是这台机器的行车记录仪。没有 root 的时候，它就是你的 root。

第一步，`atopsar -m` 拉内存曲线，找"尸体消失的时刻"：

```text
21:18:52    memfree  2126M
21:19:02    memfree 20114M    ← 空闲内存 +18GB 跳变
```

第二步，回放这个 10 秒窗口，按 RSS 找进程：

```bash
$ atop -r /var/log/atop/atop_20260721 -P PRM -b 21:18:40 -e 21:19:10 | ...
21:18:52  PID=1626391  RSS_KB=17729280    # 约 17GB，与跳变吻合
```

第三步，用 PRG 记录拿完整命令行和退出状态：

```text
21:18:52  PID 1626391 (claude) 状态 D
  cmdline: ugrep -G --ignore-files --hidden -I ... -o -E
           '.{0,150}(关键词A|关键词B)[^\\]{0,200}' /tmp/xxx/ctx_123.txt

21:19:02  PID 1626391 状态 E  exit=265    # 265 & 0x7f = 9，SIGKILL
  父进程 bash 退出码 137                   # 128 + 9，实锤被 kill -9
```

凶手找到了：一条 `ugrep` 正则搜索，21:18:56 被 SIGKILL 处决（`last` 显示管理员 21:07 用 root 登录，人工手刃）。它来自我自己挂在这台机器上的一个 AI 编码代理（Claude Code）批量日志审计任务——在导出的日志文本里检索两个关键词，顺带抓取前后各一两百个字符的上下文。

一个 grep 而已，怎么会吃 17GB？

## 四、反转二：文件只有 11KB

我的第一反应是：目标文件一定巨大，或者是没有换行的超长单行，把行缓冲撑爆了。

```bash
$ ls -la /tmp/xxx/ctx_123.txt
-rw-r--r-- 1 ... 11464 ... ctx_123.txt        # 11KB
```

**11KB。** 一个正常人类眼睛一秒扫完的文件。

再看它的内存增长曲线：

```text
20:01:12  RSS = 1.5 GB     # 启动后 10 秒
20:10:02  RSS = 15.2 GB    # 9 分钟冲到 15GB
21:18:02  RSS = 17.3 GB    # 78 分钟，一个匹配结果都没输出
```

前 9 分钟狂吃 15GB，之后匀速阴燃。这个形状说明它根本没在"搜索"——**它卡在正则编译阶段就没出来过**。问题不在文件，在这条正则：

```regex
.{0,150}(关键词A|关键词B)[^\\]{0,200}
```

注意时间线：20:01 进程启动，**20:03 整机 PSI 就打满了**。从点火到全机瘫痪，只用了两分钟。

## 五、反转三：你敲的是 grep，跑的是 ugrep

atop 里有个诡异细节：进程的 comm 是 `claude`，命令行却是 `ugrep`，而审计任务原始命令写的明明是 `grep`。三个名字，一个进程。

翻开 Claude Code 会话的 shell 快照，真相是一个影子函数：

```bash
grep () {
    ...
    # 用 claude 二进制内嵌的 ugrep 替身执行
    ( exec -a ugrep "$_cc_bin" -G --ignore-files --hidden -I \
        --exclude-dir=.git ... ${1+"$@"} )
}
```

Claude Code 会在它的 Bash 会话里把 `grep`、`find` 影子（shadow）成自带的 ugrep/bfs——好处是行为跨平台一致、默认忽略 `.git`；代价是：你以为在用 GNU grep，实际在用一个**引擎完全不同**的实现。

- **GNU grep**：惰性 DFA + 回溯混合，状态按需构造，有界量词几乎不额外花钱；
- **ugrep**：基于 RE/flex 的 **DFA 全展开**编译。Unicode 模式下 `.` 是"任意 UTF-8 字符"这个庞大字符类，`.{0,150}` 意味着把它拷贝展开 150 层再做子集构造，后面的 `[^\\]{0,200}` 再来 200 层——状态数组合爆炸。

同一条命令，两个世界：

```bash
# GNU grep：0.088 秒，找到 2 处匹配
$ time /usr/bin/grep -c -o -E '.{0,150}(关键词A|关键词B)[^\\]{0,200}' ctx_123.txt
2
real    0m0.088s

# 内嵌 ugrep：限 2GB 虚拟内存复现，20 秒内 core dump
$ timeout 20 bash -c 'ulimit -v 2097152; exec -a ugrep <claude二进制> \
    -G -o -E ".{0,150}(关键词A|关键词B)[^\\\\]{0,200}" ctx_123.txt'
timeout: the monitored command dumped core
```

**爆炸只跟正则有关，跟文件大小无关。** 哪怕目标是个空文件，它照样把内存吃穿。

这不是我一个人踩的坑。Claude Code 官方仓库里躺着一排同款 open issue，其中一个连数字都和我对得上——"11KB 文件、14~16.6GB RSS"：

- [#78861](https://github.com/anthropics/claude-code/issues/78861)：Bundled ugrep balloons to 14–16.6 GB RSS on bounded-repeat patterns that GNU grep handles in 3 MB
- [#74143](https://github.com/anthropics/claude-code/issues/74143)：grep shell shim: unbounded memory growth on ERE bounded quantifiers `{m,n}`
- [#76056](https://github.com/anthropics/claude-code/issues/76056)：grep→ugrep shim children not killed on timeout; runaway memory

截至写这篇文章时（Claude Code 2.1.197），三个 issue 全部还是 open。**升级版本不能假设问题已修复。**

## 六、反转四：超时开了枪，但没打中

新的疑问：Claude Code 的 Bash 工具是有超时的，到点会终止命令。这条 grep 凭什么跑了 78 分钟？

我最初的猜测是"SSH 断开带走了看门人"——错了。真实时间线（结合本地排查和 atop 的进程家谱）是这样的：

```text
20:01:03  ugrep 启动，开始吃内存
20:03     整机 PSI 打满，我的 user slice 进入深度节流
约 20:06  Bash 工具超时触发，SIGTERM 发向外层 shell
          → 外层收到了，但 exec 链里的 ugrep 没有被一起清理
          → bash + ugrep 被 reparent 到 PID 1，成为孤儿
20:12     我挂了近 10 小时的 SSH 会话被资源风暴压死（受害者，不是原因）
20:17     重连，撑了 2 分钟又被踢；20:44 再连，卡到 21:13 集体阵亡
20:06-21:18  孤儿 ugrep 深陷 D 状态（direct reclaim 中），
             无人看管，继续阴燃 72 分钟
21:18:56  root 出手，SIGKILL
```

**超时机制其实开了枪，但没打中要害。** 这正是上面 issue #76056 描述的缺陷：shim 的子进程在超时时不会被随队清理。而 D 状态的进程连 SIGKILL 都要等它从内核态醒来才处决得掉，SIGTERM 更是挠痒痒。

教训因此也要修正：光把会话放进 tmux 保住看门人是不够的（这次看门人一直活着），**关键是重型命令自己要带枷锁**——`timeout` 管时间，`ulimit -v` 管空间，两个都是内核层面的强制，不依赖任何用户态看门人活着。

## 七、反转五：限额不但在，还触发了两千三百万次

最后一个问题最有意思。这台机器每个用户是有内存限额的，我的 slice 配的是 30GiB。防线明明存在，为什么没拦住？

看一眼 cgroup 的配置和事件计数：

```bash
$ systemctl show user-<uid>.slice -p MemoryHigh,MemoryMax
MemoryHigh=32212254720     # 30GiB，软限额
MemoryMax=infinity         # 硬限额：没有

$ cat /sys/fs/cgroup/user.slice/user-<uid>.slice/memory.events
high     23294093           # 软限额节流：触发了 2300 万次
oom      0
oom_kill 0                  # 杀进程：0 次
```

真相是：**防线不但在，还一直在拼命工作——但它的工作方式是"节流"，不是"击毙"。**

cgroup v2 的两道内存闸门行为完全不同：

- `memory.high`（软限额）：超过后，内核把这个 cgroup 里所有要分配内存的进程拖进直接回收，强行减速——**但绝不杀进程**；
- `memory.max`（硬限额）：超过且回收无果时，在 cgroup 内部触发 OOM kill，把凶手就地正法。

这台机器只配了前者。于是 ugrep 吃到限额后，得到的惩罚是"每次分配都要先帮内核搬砖回收"——它不在乎，它是个死循环，慢一点无所谓。而同一个 slice 里我的其他进程（包括每一次新 SSH 登录的 shell）全部被拖进同一个泥潭。2300 万次节流，24 核里 15 个核陷在内核态，就是这么来的。

更扎心的是，事后盘点我自己名下的负载，ugrep 只是最大的一头，不是唯一的一头：

- 十几个跑了几天甚至几周的旧 Claude 会话，合计吃着约 9 个核的 CPU；
- 一台早前 CI 测试遗留的临时 MySQL（datadir 扔在 /tmp），单独占着 1~2.4 个核，跑了不知道多少天——root 清场时它也一并被 SIGKILL 了；
- 全部加起来：**RSS 超过 35GB、CPU 约 13 个核**，全在一个"只节流不杀"的 slice 里互相踩踏。

这就是共享大内存机器的经典困境的完整版本：**只有软限额的隔离，会把"杀死一个进程"变成"折磨整台机器"。** 单个失控进程既吃不满整机（触发不了全局 OOM），也死不掉（软限额只节流），最后所有人一起在 direct reclaim 里泡着——直到 root 人肉出手。

## 八、方法论一：资源瓶颈的快速分诊

把这次绕的弯路总结成排查小抄。核心原则：**先用 PSI 分诊，再用专项工具确诊**，别一上来就扎进 top 里看 CPU。

```text
第 0 步（10 秒）：分诊
  head /proc/pressure/{cpu,memory,io}
  → 哪个文件的 some/full 高，瓶颈就是哪种资源
  → full 高 = 整机瘫痪级；some 高 full 低 = 部分任务受阻
  → 没有 PSI 的老内核：vmstat 1 看 r/b 列 + si/so + us/sy 比例

第 1 步：确诊（按分诊结果深入）
  内存 → free -h 看 available；ps --sort=-rss 找大户；
         cat /sys/fs/cgroup/.../memory.events 看是谁的限额在打
  CPU  → pidstat 1 找用户态大户；%sys 远大于 %usr 时警惕：
         多半是内存回收/自旋锁，病根不在 CPU
  IO   → iostat -x 1 看 %util 和 await；iotop 找进程

第 2 步：事后取证（进程已经死了/机器已经恢复）
  atopsar -B -r /var/log/atop/atop_YYYYMMDD   # 回放全天 PSI
  atopsar -m -r ...                            # 内存曲线找跳变时刻
  atop -r ... -P PRM/PRG -b <时刻>             # 定位进程、命令行、死因
  → atop 历史文件 world-readable，无 root 也能全程取证
```

三个反直觉的点，值得单独记住：

1. **load average 不指认资源**。D 状态进程把负载推高，但可能一个 CPU 周期都没用。load 56 的机器，瓶颈可以是内存、IO、NFS 卡死，唯独不一定是 CPU。
2. **%sys 高不等于 CPU 瓶颈**。内核态 CPU 吃满，大概率是在替别的资源打工——direct reclaim（内存）、自旋等锁、软中断（网络）。看到 %sys 高，第一件事是查内存和 IO。
3. **"SSH 登不上"本身就是资源信号**。认证过了但 shell 出不来，说明登录链路里某一环在等资源——而新会话恰好会被 pam_systemd 塞进你自己的 cgroup。如果换个用户能登上，基本可以锁定是你自己的 slice 出了问题。

## 九、方法论二：60 秒定位手册（值班视角）

如果你是那个 21:07 登上来救火的人，按这个顺序走，60 秒内能拿到凶手的完整画像：

```bash
# ① 整体态势（10 秒）：时间、负载、内存、趋势、哪个 cgroup 在吃
uptime; free -h; vmstat 1 5
systemd-cgtop -b -n 1 --depth=3

# ② 找大户（15 秒）：内存榜 + CPU 榜各扫一眼
ps -eo pid,ppid,user,stat,etime,%cpu,%mem,rss,comm,args --sort=-rss  | head -40
ps -eo pid,ppid,user,stat,etime,%cpu,%mem,rss,comm,args --sort=-%cpu | head -40

# ③ 锁定 PID 后，看它到底是谁、在干什么（20 秒）
pid=<可疑PID>
pstree -aps $pid                        # 家谱：谁生的它
readlink -f /proc/$pid/exe              # 真实二进制（伪装名无所遁形）
tr '\0' ' ' </proc/$pid/cmdline; echo   # 完整命令行
cat /proc/$pid/smaps_rollup             # 内存构成：堆? 匿名页? 共享?

# ④ 顺手看 D 状态进程都卡在哪个内核函数上
ps -eo pid,user,stat,wchan:32,etime,args | awk '$3 ~ /^D/'
```

止血时的纪律，比手速重要：

```bash
# 先礼后兵：TERM → 等 3 秒 → 还活着再 KILL
kill -TERM $pid; sleep 3
kill -0 $pid 2>/dev/null && kill -KILL $pid
```

**绝对不要图省事 `pkill <程序名>`。** 共享机上同名进程可能属于十几个用户、几十个正常会话，按名字批杀等于二次事故。杀之前必须用 ③ 确认过完整命令行，只杀确认过的 PID。

## 十、复盘：三层防线

这次事故是四层漏洞的叠加（病态正则 × 影子替换 × 超时不清理子进程 × 只软不硬的限额），收敛成三层防线：

**第 1 层：正则纪律（直接原因）**
- 有界量词 `{m,n}` 上限超过 20 的正则，强制 `command grep`（绕过影子函数走 GNU grep）或 `rg`；
- "关键词 ± 上下文"需求两步走：先 `grep -n` 拿行号，再脚本按偏移切片。别用 `.{0,N}关键词.{0,M}` 一发入魂；
- 预期超 1 分钟的重型命令，主动包 `timeout` + `ulimit -v`——内核层面的枷锁，不依赖任何用户态看门人。

**第 2 层：会话卫生（放大器）**
- 给 AI 代理会话套上 cgroup 硬限额，单个失控子进程最多炸掉自己的 scope，拖不垮整机：

```bash
systemd-run --user --scope \
  -p MemoryHigh=6G -p MemoryMax=8G \
  claude
```

- 定期清点自己名下的进程（`ps -u $USER --sort=-rss`），跑完的旧会话、临时数据库及时关掉。这次盘点前，我名下挂着 115 个进程、20GB RSS——每一个都是"当时忘了关"；
- 临时 MySQL 之类的测试守护进程，用完 `mysqladmin shutdown`，别让它带着 /tmp 里的 datadir 陪跑几周。

**第 3 层：系统兜底（最后防线）**
- 用户级：5 分钟一跑的 cron 巡检——可用内存低于阈值、或单进程 RSS 超阈值就发 IM 告警，消息里直接带 atop 回放命令。这次要是有它，20:10 就该有人收到消息，而不是 78 分钟后靠 root 巡逻；
- root 级：**软限额必须配硬限额**。`MemoryHigh` 之外再设 `MemoryMax`（比如 high 的 1.2 倍），让失控进程在拖垮邻居之前被 cgroup OOM 就地击毙。"节流不杀"的仁慈，对死循环就是纵容。

## 尾声

巡检脚本上线那天，我给告警文案加了一行处置提示：

> atop -r /var/log/atop/atop_$(date +%Y%m%d) 回放定位

毕竟这次能破案，全靠 atop 这台默默转了 25 天的行车记录仪，和三个 10 秒就能读完的 PSI 文件。
